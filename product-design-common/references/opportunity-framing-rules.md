# Opportunity Framing Rules

Frame outcomes before features.

Good opportunity framing:
- names the target user and situation
- solves a whole problem or meaningful slice, not only a UI surface
- states desired outcome and success measure
- separates pains/opportunities from solution hypotheses
- uses evidence or marks assumptions
- considers feasibility, privacy, accessibility, and trust risks
- picks the smallest useful concept/test

Bad framing:
- starts with “add dashboard” or “make it AI”
- copies competitor features without user need
- optimizes for aesthetics before task clarity
- ignores missing data, integration limits, or privacy risk

Output should include validation tests that can disprove the concept.
